# Contributing

## Before opening an issue or pull request

- Explain the user problem and the expected product-design behavior.
- Keep examples generic and remove client, personal, and internal project information.
- Preserve the clarification gate, confirmation gates, output safety rules, and version history.
- Validate changed plugin manifests and skills before submitting.

## Pull requests

Describe the changed workflow, affected files, compatibility considerations, validation performed, and any migration impact. Do not include generated project outputs, private knowledge snapshots, local backups, or credentials.

## Design changes

Changes to UI guidance should state the external source or design-system rationale. Changes to mandatory workflow rules should explain the user-facing behavior and confirmation impact.
